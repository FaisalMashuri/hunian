Fomo-style model (Rp50k subscription for aggregated rental market intelligence) analyzed vs existing one-time per-cycle model (Rp49-79k).

**Decision: Keep one-time per-cycle. Collect negotiation/all-in data PASSIVELY as post-decision "tutup siklus" feature. Defer monetizing intelligence as separate product.**

**Why:** Cold-start timing inversion is nearly fatal for solo dev with no budget. Contribute-consume mismatch (users need data pre-decision, can only contribute post-decision) breaks the core Fomo mechanic. Charging for sparse data → churn → reputation damage.

**The valid insight extracted from this analysis:** Negotiation outcome data + all-in cost breakdown IS genuine whitespace that no competitor has. Mamikos is *structurally unable* to collect negotiation data (would weaken landlord position = alienate supply side). This is a structural moat from competitor conflict of interest.

**How to apply:** When monetization alternatives come up, treat the data flywheel as a FEATURE enhancement path, not a separate product. The sequence: (1) Collect passively via post-decision "Catat Hasil Final" modal (3 optional fields, framed as personal record not community contribution); (2) When density hits ~30 data points per area/type cell in 2-3 cities (~12-18 months), integrate as feature: "listing ini 15% di atas rata-rata area Menteng Dalam dari 47 transaksi nyata"; (3) Only then consider B2B or Pro tier monetization.

**What NOT to build now:** Market intelligence dashboard as standalone subscription product.

**Strategic risk to monitor:** Well-funded competitor (Mamikos or new startup) could enter negotiation data whitespace. Mitigation: build contribution habit early without monetization pressure — historical data cannot be retroactively duplicated.
